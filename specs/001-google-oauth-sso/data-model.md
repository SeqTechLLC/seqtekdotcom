# Phase 1 Data Model — Google Workspace SSO for `/admin`

**Feature**: 001-google-oauth-sso · **Date**: 2026-05-21

Two Payload collections changed (one extended, one new) plus one stdout-only event
shape used for CloudWatch auditing. Schema is generated by Payload — the migration is
auto-derived from the collection configs; no hand-written DDL.

---

## 1. `users` (extended)

The existing `users` collection (currently `auth: true` only — Payload's default
email/password) is wrapped with `withUsersCollection` from `payload-auth-plugin` and
extended with role + display-name fields. Local strategy is disabled.

| Field                         | Type                   | Required | Notes                                                                                                                                                                                                                                                                                                                           |
| ----------------------------- | ---------------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `id`                          | uuid (Payload-managed) | yes      | PK.                                                                                                                                                                                                                                                                                                                             |
| `email`                       | text                   | yes      | Indexed, unique. Sourced from Google `email` claim on first OAuth sign-in.                                                                                                                                                                                                                                                      |
| `name`                        | text                   | yes      | Sourced from Google `name` claim. Used as `useAsTitle` in admin UI.                                                                                                                                                                                                                                                             |
| `roles`                       | select (multi)         | yes      | Options: `admin`, `editor`. Default `editor`. Bootstrap-Admin hook overrides to `admin` if the table contains zero admins at the time of insert. Matches ARCHITECTURE.md §6 Access Control matrix.                                                                                                                              |
| `createdAt`                   | timestamp              | yes      | Payload-managed.                                                                                                                                                                                                                                                                                                                |
| `updatedAt`                   | timestamp              | yes      | Payload-managed.                                                                                                                                                                                                                                                                                                                |
| _plugin-injected auth fields_ | various                | —        | `withUsersCollection` adds `hashedPassword`, `hashSalt`, `hashIterations`, `verificationCode`, `verificationHash`, `verificationTokenExpire`, `verificationKind`, `claims`. These exist in the schema but are unused because `disableLocalStrategy: true` + no `PasswordProvider` registered means no code path writes to them. |

**Config flags**:

- `auth.disableLocalStrategy: true` — no `/api/users/login` endpoint.
- `auth.tokenExpiration` — unset (Payload default = 7200 s; see research R-3).
- `admin.useAsTitle: 'name'` (falls back to `email` if name is blank).

**Hooks** (defined in `src/collections/Users.ts`):

| Hook           | Trigger                  | Behaviour                                                                                                                                                                                                                                                                                                                                                                                  |
| -------------- | ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `beforeChange` | `operation === 'create'` | (1) If `req.user == null` (OAuth auto-provision path), enforce `isWorkspaceEmail(data.email)`; on failure call `logSignIn({ outcome: 'domain-rejected', ... })` then throw `ValidationError`. (2) If `req.user == null`, count current admins; set `data.roles = ['admin']` when zero admins exist, else `['editor']`. (3) Always call `logSignIn({ outcome: 'success', ... })` on accept. |
| `beforeChange` | `operation === 'update'` | If `req.user == null`, forbid role changes (`data.roles`). Only an authenticated admin can promote/demote (FR-007/FR-008).                                                                                                                                                                                                                                                                 |

**Access control** (preserves ARCHITECTURE.md §6 matrix):

```text
read    → ({ req: { user } }) => Boolean(user)
create  → () => false                          // only the OAuth flow creates users
update  → ({ req: { user } }) => user?.roles?.includes('admin')
delete  → ({ req: { user } }) => user?.roles?.includes('admin')
admin   → ({ req: { user } }) => Boolean(user?.roles?.length)  // /admin gate
```

## 2. `accounts` (new)

A `withAccountCollection`-wrapped collection that holds the OAuth provider binding
per identity. One user → many accounts (in our case always 1:1 with Google for
phase 1, but the shape supports multi-provider linkage later without schema change).

| Field                     | Type                   | Required | Notes                                                                                                                                               |
| ------------------------- | ---------------------- | -------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| `id`                      | uuid                   | yes      | PK.                                                                                                                                                 |
| `user`                    | relationship → `users` | yes      | Many-to-one. Indexed.                                                                                                                               |
| `issuerName`              | text                   | yes      | `'google'` for this feature. Indexed (composite with `sub` for lookup).                                                                             |
| `sub`                     | text                   | yes      | Google's stable subject ID. Composite-unique with `issuerName`. **Load-bearing for FR-006.**                                                        |
| `name`                    | text                   | no       | Mirror of Google `name` claim at last sign-in.                                                                                                      |
| `picture`                 | text                   | no       | Mirror of Google `picture` claim. Not rendered, but kept for parity with plugin defaults.                                                           |
| `scope`                   | text                   | no       | OAuth scope granted.                                                                                                                                |
| `access_token`            | text                   | no       | Per-plugin storage. Not used by application code; the plugin handles refresh internally.                                                            |
| `refresh_token`           | text                   | no       | As above.                                                                                                                                           |
| `expires_in`              | number                 | no       | As above.                                                                                                                                           |
| `passkey`                 | group                  | no       | Unused (no PasskeyAuthProvider registered). Kept because the plugin's helper always injects it; pruning it diverges from upstream and breaks types. |
| `createdAt` / `updatedAt` | timestamp              | yes      | Payload-managed.                                                                                                                                    |

**Unique constraint**: `(issuerName, sub)` — added in the auto-generated migration.
This is what makes "same Google identity across email changes" idempotent (FR-006).

**Access control**: all operations gated to `req.user?.roles?.includes('admin')`. The
admin panel exposes the collection so an admin can revoke a binding manually if
needed (break-glass exception to "Break-glass is out of scope" — listing rows and
deleting is built-in UI, no code added).

## 3. Sign-in audit event (stdout / CloudWatch — not a collection)

Per research R-6, audit entries are JSON lines on stdout. No DB schema. The shape:

```jsonc
{
  "event": "admin_sign_in",
  "ts": "2026-05-21T14:32:18.220Z", // ISO 8601
  "email": "user@seqtechllc.com", // lowercased
  "outcome": "success", // 'success' | 'domain-rejected' | 'oauth-error'
  "provider": "google",
  "ip": "203.0.113.10", // optional; from req.headers.get('x-forwarded-for') first hop
  "userId": "9f4c…", // present only on outcome=success
  "errorCode": "state_mismatch", // present only on outcome=oauth-error
}
```

**Producer**: `src/lib/auth/sign-in-audit.ts` exports `logSignIn(input)`. Called by
the Users `beforeChange` hook (success + domain-rejected) and by the
`/admin/login?error=...` server component (oauth-error) if we render one.

**Consumer**: CloudWatch Logs (existing log group for the EC2 container — see
ARCHITECTURE.md §6.1 / Container Strategy). Queryable via CloudWatch Insights:

```text
fields @timestamp, email, outcome, errorCode
| filter event = 'admin_sign_in' and outcome != 'success'
| sort @timestamp desc
| limit 50
```

## 4. State transitions

### 4a. First OAuth sign-in (auto-provision)

```text
[no users row] ──Google callback──> [domain check]
                                       │
                  domain != seqtechllc.com
                                       ▼
                                   ValidationError + audit('domain-rejected')
                                       │
                  domain == seqtechllc.com
                                       ▼
                          [count(role=admin)]
                              │             │
                            0 │             │ >=1
                              ▼             ▼
                  users.create(role=admin) users.create(role=editor)
                              │             │
                              ▼             ▼
                       accounts.create(sub, issuerName='google', user=<new id>)
                              │
                              ▼
                       audit('success') + 302 /admin
```

### 4b. Returning OAuth sign-in

```text
[existing accounts row matching (issuerName='google', sub=<google sub>)]
                              │
                              ▼
              plugin issues session JWT against accounts.user
                              │
                              ▼
                       audit('success') + 302 /admin
```

The users row's `role` is **not** re-derived on returning sign-ins — preserved
exactly as last edited by an admin (FR-007).

### 4c. Role change

```text
admin opens /admin/collections/users/<id>
   │
   ▼
edits roles select (editor ↔ admin)
   │
   ▼
PATCH /api/users/<id>  (cookie-authenticated; req.user is the editing admin)
   │
   ▼
Users beforeChange (operation='update'):
   req.user.roles includes 'admin' → allow
   req.user is null                → ValidationError (covers stolen-cookie edge)
   │
   ▼
DB row updated; next sign-in reflects new role
```

## 5. Migration strategy

- One auto-generated migration is committed to `src/migrations/` (filename
  `YYYYMMDDHHMMSS_001_oauth_sso.ts`) capturing: column additions to `users` (`name`,
  `roles`), creation of the `accounts` table, and the composite unique index.
- Cutover starts from a fresh `users` table (Clarifications 2026-05-21 Q2). The
  migration runs cleanly on an empty table. The spike-era `kenn` row and any
  spike-authored content are discarded as a precondition, not by this migration.
- No data backfill — there is no prior `accounts` data to migrate.
