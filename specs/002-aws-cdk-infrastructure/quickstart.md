# Quickstart: Deploying SEQTEK Website Infrastructure

**Audience**: An engineer (you, future-you, or a future contributor) standing up the SEQTEK website infrastructure from a fresh clone. The goal is **a deployed staging environment in under 60 minutes** (SC-001 scope-extended to staging — same procedure as prod minus the DNS step).

If you're already past the bootstrap and just deploying changes, skip to §6 "Routine deploys."

---

## 0. Prerequisites checklist

You need, before starting:

- [ ] AWS account access (existing SEQTEK account; account ID will be picked up via `aws sts get-caller-identity`)
- [ ] AWS CLI v2 installed locally; `aws configure sso` or equivalent producing valid credentials
- [ ] Node.js ≥ 22 + npm (already a project requirement)
- [ ] Docker Desktop or compatible (you do not need this for `cdk deploy` — only for local image builds and `docker push` to ECR during the first manual deploy)
- [ ] A Google Workspace OAuth client for `/admin` SSO (per spec 001; already exists for prod, you may need to create one for staging)
- [ ] Slack workspace admin access (Kenn) — to create `#seqtek-website-alerts` channel + incoming webhook

Time budget: 60 min for first-time / fresh-account; ~5 min for routine deploys.

---

## 1. One-time AWS account bootstrap

These run once per AWS account. If you're using the existing SEQTEK account and someone has already bootstrapped CDK, skip to §2.

```sh
# From repo root
cd infra
npm ci

# CDK bootstrap (creates the CDKToolkit stack — S3 bucket for assets + ECR repo for asset images + deploy IAM roles)
npx cdk bootstrap aws://<account-id>/us-east-1

# Deploy the network stack — this also creates the GitHub OIDC identity provider for the account
npx cdk deploy -c env=prod SeqtekProdNetwork
npx cdk deploy -c env=staging SeqtekStagingNetwork
```

The `SeqtekProdNetwork` stack output includes the OIDC provider ARN — note it down. You'll wire it into GitHub Environments below.

---

## 2. Seed Parameter Store with the "manual" secrets

Three values that aren't generated by CDK (Google OAuth credentials + Slack webhook URL). The CDK-generated values (DB password, Payload secret, revalidation secret) are populated by the `cdk deploy SeqtekProdData` step in §4; do not pre-populate those.

```sh
# Google OAuth client (per spec 001 — create in Google Cloud Console: Workspace project → OAuth 2.0 Client)
# Use redirect URIs:
#   prod:    https://seqtek.com/api/auth/oauth/callback/google
#   staging: https://<staging-cloudfront-domain>/api/auth/oauth/callback/google
#            (run `cdk deploy SeqtekStagingEdge` first to get the staging CloudFront URL, then come back to set the redirect)

aws ssm put-parameter \
  --name /seqtek/website/prod/google_client_id \
  --type String \
  --value '<copy from Google Cloud Console>'

aws ssm put-parameter \
  --name /seqtek/website/prod/google_client_secret \
  --type SecureString \
  --value '<copy from Google Cloud Console>'

# Repeat for staging with /seqtek/website/staging/... paths

# Slack incoming webhook (Kenn — create channel + webhook; copy URL once)
aws ssm put-parameter \
  --name /seqtek/website/prod/slack_webhook_url \
  --type SecureString \
  --value 'https://hooks.slack.com/services/T.../B.../...'

# Same for staging — you may use the same channel, or split into #seqtek-website-alerts-staging
```

Verify:

```sh
aws ssm get-parameters-by-path --path /seqtek/website/prod --recursive --query 'Parameters[].Name'
# Expected: google_client_id, google_client_secret, slack_webhook_url
# The remaining paths get created in §4.
```

## 3. GitHub repository setup

One-time per AWS account / repo combination.

1. **Repository variable**: Settings → Secrets and variables → Actions → Variables tab → add `AWS_ACCOUNT_ID` (non-secret).
2. **GitHub Environments**: Settings → Environments → create `production` and `staging`.
   - `production`: required reviewers = [the engineers authorized to ship to prod]. Wait timer optional.
   - `staging`: no required reviewers; deploys are routine.
3. **Branch protection on `main`**: Settings → Branches → require status checks (`ci.yml` jobs + `infra-diff`) before merging.

No secret needs to be created. OIDC trust is established server-side via the `SeqtekProdNetwork` stack from §1.

## 4. Deploy the data + compute + edge + observability stacks

In order — these stacks have dependencies (data ← compute, compute → edge, all → observability).

```sh
# Staging first (smaller blast radius if something's wrong)
npx cdk deploy -c env=staging 'SeqtekStaging*'

# Then prod
npx cdk deploy -c env=prod 'SeqtekProd*'
```

`cdk deploy` will:

1. Synth all matching stacks
2. Show a CloudFormation change-set preview
3. Prompt for approval on IAM changes (use `--require-approval=any-change` to elide for routine deploys — but the first time, review them)
4. Apply each stack in dependency order

Expected wall-clock: ~25 min for staging from a clean state (RDS provisioning is the long pole — ~15 min), ~25 min for prod.

After the data stack lands, verify the CDK-generated parameters arrived:

```sh
aws ssm get-parameters-by-path --path /seqtek/website/prod --recursive --query 'Parameters[].Name'
# Expected: 9 paths total (3 manual from §2 + 6 CDK-generated)
```

After the edge stack lands, note the CloudFront distribution domain (CDK output `CloudFrontDomainName`) — that's the URL to validate the deploy against.

## 5. Smoke-test the deploy

```sh
# Health probe
curl -s https://<cloudfront-domain>/api/health | jq
# Expected: { "status": "ok", "uptime": …, "db": "ok", "timestamp": "…", "responseTimeMs": <small number> }

# Public page
curl -s -o /dev/null -w '%{http_code}\n' https://<cloudfront-domain>/
# Expected: 200

# Admin SSO entry
curl -s -o /dev/null -w '%{http_code}\n' https://<cloudfront-domain>/admin
# Expected: 200 (SSO entry screen; redirect to Google is JS-driven)
```

If `/api/health` returns 503: the container is up but DB is unreachable. Check ASG instance logs in CloudWatch (log group `/seqtek/website/{env}/app`) for the DB connection error.

If `/admin` returns 502 or 504: ALB has no healthy targets. Check the ALB target group health in the AWS console — most common cause is the container failing to start (Parameter Store path missing, or Docker image fetch failed).

If the CloudFront URL itself doesn't resolve: DNS propagation. CloudFront's default distribution domain takes ~5 min to propagate after first deploy.

Trigger the Slack heartbeat test:

```sh
# Find the EventBridge rule
aws events list-rules --query "Rules[?Name=='SeqtekProdObservability-Heartbeat'].Name"

# Manually invoke it
aws events put-events --entries 'Source=manual-test,DetailType=Heartbeat,Detail={}'
```

You should see the gray ⚙️ heartbeat message in the Slack channel within ~30 seconds. If you don't: Lambda logs are at `/aws/lambda/SeqtekProdObservability-SlackNotifier*`.

## 6. Routine deploys

After bootstrap, the loop is:

- Open a PR. `infra-diff` posts the `cdk diff` as a comment within ~5 minutes.
- Review the diff. Merge to `main`.
- `deploy.yml` runs automatically: builds the Docker image, pushes to ECR, runs `cdk deploy 'SeqtekProd*Compute'`, triggers ASG instance refresh, waits for health, runs Playwright smoke. ~15 min end-to-end.
- A green deploy posts a 🟢 deployment status on the merge commit. A failed deploy reports the failure and CloudFormation rolls back automatically (FR-015).

Network / data / edge / observability changes do **not** auto-deploy on merge — they require `workflow_dispatch` against `deploy.yml` with `stack-filter` set to the target glob, and the GitHub `production` environment's required-reviewers gate gives a manual approval surface.

## 7. Verifying alarms work end-to-end

Once per environment, simulate each alarm to confirm the alert reaches Slack:

| Alarm               | How to simulate                                                                                                                                |
| ------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| AlbFiveXx           | Temporarily change a route to throw; deploy to staging; hit it 6+ times in 5 min. Revert after.                                                |
| AlbUnhealthyHost    | Stop Docker on a staging instance (`ssh + docker stop`); wait 2 min.                                                                           |
| Ec2CpuHigh          | `ssh + stress --cpu 2 --timeout 600`.                                                                                                          |
| Ec2DiskHigh         | `ssh + fallocate -l 8G /tmp/big`.                                                                                                              |
| RdsCpuHigh          | Run a sustained load query against staging DB (e.g., `SELECT pg_sleep(0.001), … FROM generate_series(1, 1000000)`).                            |
| RdsFreeStorageLow   | Threshold is 2 GB; on staging gp3 50GB, fill ~47 GB temporarily. Or temporarily lower the threshold to ~45 GB in CDK, deploy, observe, revert. |
| RdsConnectionsHigh  | Open `~ pool max × 0.8 + 1` long-lived `psql` sessions to staging DB.                                                                          |
| CloudFrontErrorRate | Deploy a deliberately-broken route, request it through CloudFront 100+ times in 5 min.                                                         |

Each simulation should produce a 🚨 Slack message within ~5 min. Document the simulation in `docs/RUNBOOK.md` as it stabilizes (RUNBOOK is out of scope for spec 002 — a Phase 5 doc).

## 8. Secret rotation procedure

Per `contracts/parameter-store.md` §Rotation matrix:

### Database password (`database_url`)

```sh
# Trigger RDS password rotation (Secrets Manager handles the actual change)
aws secretsmanager rotate-secret --secret-id <rds-master-secret-id>

# CDK detects the new value on next synth; mirror to Parameter Store:
npx cdk deploy -c env=prod SeqtekProdData

# Trigger ASG instance refresh (zero-downtime; new instances pick up new password)
aws autoscaling start-instance-refresh \
  --auto-scaling-group-name <prod-asg-name> \
  --preferences MinHealthyPercentage=100,InstanceWarmup=180
```

### Payload secret (`payload_secret`)

```sh
# Generate new secret
NEW_SECRET=$(openssl rand -hex 32)

# Update Parameter Store
aws ssm put-parameter \
  --name /seqtek/website/prod/payload_secret \
  --type SecureString \
  --value "$NEW_SECRET" \
  --overwrite

# ASG instance refresh — note: this invalidates all admin sessions
aws autoscaling start-instance-refresh …
```

### Google OAuth client secret

```sh
# Rotate in Google Cloud Console first (generates new secret, old works during transition window)

# Update Parameter Store
aws ssm put-parameter \
  --name /seqtek/website/prod/google_client_secret \
  --type SecureString \
  --value '<new value>' \
  --overwrite

# ASG instance refresh
aws autoscaling start-instance-refresh …
```

### Slack webhook URL

```sh
# Generate new webhook in Slack workspace settings (revoke old)
aws ssm put-parameter \
  --name /seqtek/website/prod/slack_webhook_url \
  --type SecureString \
  --value 'https://hooks.slack.com/services/T.../B.../<new>' \
  --overwrite

# NO ASG refresh needed — Lambda re-fetches webhook URL on next cold start.
# To force immediate adoption, bump the Lambda's env vars to invalidate the cache.
```

## 9. Tear-down

```sh
# Staging
npx cdk destroy -c env=staging 'SeqtekStaging*'

# Prod (be very sure)
npx cdk destroy -c env=prod 'SeqtekProd*'
```

CDK will:

- Refuse to delete the RDS instance if `deletionProtection: true` (prod). Override by deploying once with `deletionProtection: false`, then destroying.
- Refuse to delete the S3 media bucket if it contains objects. Empty it first: `aws s3 rm s3://seqtek-media-prod --recursive` (irreversible; export to a backup first if media is precious).
- Delete CloudFormation stacks in reverse-dependency order automatically.

Verify zero billing residue:

```sh
# Wait ~24h, then:
aws ce get-cost-and-usage \
  --time-period Start=$(date -u -d 'yesterday' +%Y-%m-%d),End=$(date -u +%Y-%m-%d) \
  --granularity DAILY \
  --metrics BlendedCost \
  --filter '{"Tags":{"Key":"Project","Values":["seqtek-website"]}}'
# Expected: $0.00 per day after teardown (FR-025, SC-008)
```

## 10. Troubleshooting

### `cdk deploy` hangs at "CREATE_IN_PROGRESS" for >20 min

Almost always RDS provisioning, which is slow. Check CloudFormation events in the console. If a different resource is stuck, capture the resource ID and check its service-specific console — most resources have informative status reasons.

### `cdk diff` shows a destructive change you didn't expect

Stop. Do not apply. Investigate:

- A property change marked `(may cause replacement)` triggers a destroy+create. For stateful resources (RDS, S3) this is data loss.
- Recent CDK version bumps occasionally change a default; check the changelog.
- If the change is legitimate, do it via `workflow_dispatch` with explicit approval, not `cdk deploy` from a laptop.

### Instance refresh stuck or rolling back

```sh
aws autoscaling describe-instance-refreshes \
  --auto-scaling-group-name <asg-name> \
  --query 'InstanceRefreshes[0].{Status:Status,Reason:StatusReason,PercentageComplete:PercentageComplete}'
```

If `Status: Failed`: the new instances aren't passing health checks. Cancel the refresh, check instance logs.

### Concurrent deploy collision

`gh workflow run deploy.yml` while another deploy is in flight will queue (concurrency group). CloudFormation also locks the stack. Wait for the in-flight deploy to finish (~15 min); the queued run will then start automatically. No corruption is possible (FR-016).

## Verification checklist (use this for SC-001 timing)

When standing up a fresh environment, tick these as you go. The whole list should fit in under 60 min for staging.

- [ ] AWS CLI authenticated (`aws sts get-caller-identity` returns the SEQTEK account)
- [ ] `cdk bootstrap` complete (CDKToolkit stack visible in CloudFormation console)
- [ ] `SeqtekStagingNetwork` deployed; OIDC provider visible in IAM
- [ ] Parameter Store seeded with `google_client_id`, `google_client_secret`, `slack_webhook_url` for staging
- [ ] `SeqtekStaging*` stacks all deployed (data, compute, edge, observability)
- [ ] `/api/health` returns 200 on staging CloudFront URL
- [ ] `/admin` reaches SSO entry on staging CloudFront URL
- [ ] Heartbeat appears in Slack channel within 6h (or trigger manually for immediate verification)
- [ ] CDK assertion tests green: `npm --prefix infra run test`
- [ ] Tear-down dry-run: `cdk destroy --dry-run -c env=staging 'SeqtekStaging*'` — no surprises

If you blow the 60-min budget, capture where time went and amend this quickstart in the same PR — the budget exists to keep the docs honest.
