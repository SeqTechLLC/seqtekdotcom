# Contract: Parameter Store Namespace

**Prefix**: `/seqtek/website/{env}/` (env ∈ {`prod`, `staging`})

All runtime secrets and per-env configuration values live under this prefix. EC2 instance profile + Slack Lambda role are ARN-scoped to this prefix (`arn:aws:ssm:us-east-1:<acct>:parameter/seqtek/website/{env}/*`). No application code reads any Parameter Store path outside this prefix.

## Path inventory

| Path                     | Type           | Provisioner                                 | Consumer                                | Notes                                                                                                                                                                                       |
| ------------------------ | -------------- | ------------------------------------------- | --------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `…/database_url`         | `SecureString` | CDK (`data-stack`)                          | App (Payload DB adapter)                | Format: `postgresql://<user>:<pass>@<host>:5432/seqtek_{env}`. ## Provisioner semantics

- **CDK (write)**: Created/updated as part of `cdk deploy`. Engineers do not edit these out-of-band — drift gets corrected on next deploy. CDK reads via `StringParameter.fromStringParameterName()` at runtime for cross-stack references.
- **Manual (write)**: Seeded by a human via AWS Console, AWS CLI, or `aws ssm put-parameter`. The first-deploy checklist in `quickstart.md` enumerates each of these with the exact CLI command.

## Application read path

- Container start-up: the `LaunchTemplate` user-data script runs `aws ssm get-parameters --names /seqtek/website/{env}/database_url …` and exports each as an environment variable before `docker run`.
- Runtime credential rotation: changing a SecureString does **not** auto-propagate into a running container's env. Procedure: rotate the parameter → trigger ASG instance refresh → new instances pick up the new value. Documented in quickstart.md §secret rotation.
- The app's `process.env` reads remain the existing names (`DATABASE_URL`, `PAYLOAD_SECRET`, etc.) — Parameter Store is a packaging detail, not an API change for app code.

## What is NOT in this namespace

- `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` — never present anywhere. EC2 reaches AWS APIs via instance profile, Lambda via execution role. (Constitution IV.)
- `NEXT_PUBLIC_*` build-time variables (`NEXT_PUBLIC_HUBSPOT_PORTAL_ID`, `NEXT_PUBLIC_GTM_ID`, `NEXT_PUBLIC_SCOREAPP_URL`) — these are baked into the container image at `docker build` time as build args. Sourced from GitHub Actions repository variables, not Parameter Store, because they need to exist at build time before the container is even pushed to ECR.

## Rotation matrix

| Parameter                          | Procedure                                                                                                                                   | Customer impact                 |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------- |
| `database_url` (password rotation) | Update RDS master password → CDK detects drift in Secrets Manager → next `cdk deploy` mirrors to SSM → ASG instance refresh                 | None (refresh is zero-downtime) |
| `payload_secret`                   | Update SSM → ASG instance refresh → **all sessions invalidated** (next request requires re-login)                                           | Admin users re-login            |
| `google_client_secret`             | Generate new in Google Cloud Console → update SSM → ASG instance refresh                                                                    | None                            |
| `revalidation_secret`              | Update SSM → ASG instance refresh                                                                                                           | None (revalidation is internal) |
| `slack_webhook_url`                | Generate new in Slack → update SSM → Lambda picks up on next invocation (no Lambda redeploy needed — Lambda reads parameter per-invocation) | None                            |

## IAM policy summary

```ts
// AppInstanceRole — EC2 instance profile
new iam.PolicyStatement({
  actions: ['ssm:GetParameters', 'ssm:GetParameter'],
  resources: [`arn:aws:ssm:us-east-1:${account}:parameter/seqtek/website/${env}/*`],
})

// SlackNotifierRole — Lambda
new iam.PolicyStatement({
  actions: ['ssm:GetParameter'],
  resources: [`arn:aws:ssm:us-east-1:${account}:parameter/seqtek/website/${env}/slack_webhook_url`],
})
```

No `ssm:*`. No wildcard resource. Assertion-tested in `infra/test/iam-invariants.test.ts`.
